Design principle
Two channels, two jobs — never crossed
The QR code is an optical air-gap channel — physically secure by nature, slow by nature. It carries only the tiny key material needed to establish trust.

WebRTC DataChannel is a high-speed data channel — fast enough for large files, but needs a secure way to bootstrap. It carries only the encrypted file, never any keys.

The two never overlap. By the time file bytes flow, the QR is done and can be dismissed.
Complete session flow — step by step
Phase 1 — QR key exchange (optical, air-gapped)
1
Sender generates ephemeral ECDH key pair
Web Crypto API: generateKey({name:"ECDH", namedCurve:"P-256"}). Private key never leaves the browser tab. Public key is 65 bytes uncompressed.
2
Sender encodes bootstrap payload into QR
Payload is ~180 bytes — fits comfortably in QR v12. Rotates every 8 seconds with a fresh HMAC-SHA256 timestamp to prevent replay. Displayed full-screen.
3
Receiver scans QR with camera
Parses payload, validates HMAC + timestamp (rejects if >15s old). Extracts sender's public key and session ID. Generates its own ECDH key pair immediately.
4
Both sides derive the shared secret independently
ECDH: sender's private key × receiver's public key = receiver's private key × sender's public key. Shared secret hashed through HKDF-SHA256 → 256-bit AES-GCM key. Neither side ever transmits the key — it's derived from math.
Phase 2 — WebRTC signaling (server-blind)
5
Receiver connects to signaling server via WSS
Sends: {sessionID, receiverPubKey} — the receiver's public key rides here so the sender can complete ECDH. Signaling server sees only the session ID and opaque blobs.
6
Sender gets notified, completes ECDH
Sender was polling signaling by sessionID. Gets receiver's pubKey, derives the same AES-GCM key locally. Both sides now share the key without ever transmitting it.
7
SDP offer / answer exchange via signaling
Standard WebRTC negotiation. SDP blobs are encrypted with the shared AES-GCM key before being sent through the signaling server — so even the SDP is opaque to the server.
8
ICE candidates exchanged, DTLS handshake completes
Direct P2P path found via STUN. DTLS-SRTP negotiated automatically by the browser. DataChannel opens. Signaling server is now idle — session purged.
Phase 3 — encrypted file transfer (pure P2P)
9
Sender sends metadata frame first
Encrypted with AES-256-GCM: {filename, size, type, totalChunks, sha256}. Receiver shows "Accept transfer?" dialog before any file data arrives.
10
Chunked file transfer over DataChannel
64 KB chunks, each encrypted independently with AES-256-GCM + unique 96-bit IV. Sender watches bufferedAmount for flow control. Receiver buffers to IndexedDB.
11
SHA-256 integrity check + download
Receiver hashes reassembled plaintext. Must match sender's hash from step 9. On match → download triggered. On mismatch → alert user, discard, request resend.
QR payload — exact binary layout
Total: ~178 bytes. Fits in QR version 12, binary mode, ECC level M. Scans reliably at 20–30 cm on any modern phone.
[2] magic: 0x51 0x52 ("QR")
[1] version: 0x01
[16] session_id: CSPRNG 128-bit random
[65] sender_pub: ECDH P-256 public key (uncompressed)
[8] timestamp: Unix ms (uint64, big-endian)
[4] ttl: 15000 (15 seconds, uint32)
[50] sig_server: WSS signaling server URL (ASCII, null-padded)
[32] hmac: HMAC-SHA256(all above fields, session_secret)
The session_secret used for HMAC is derived from the ECDH private key + timestamp — so only the true sender can produce a valid HMAC. Receiver verifies before proceeding.
Encryption layers — what's protected where
QR channel
Physical air-gap
Only the person holding the camera can read it. No network exposure.
HMAC-SHA256 signature
Proves QR was generated by the true sender. Timestamp prevents replay.
5s visual rotation
Shoulder-surf / screenshot window is tiny.
WebRTC channel
TLS 1.3 on WSS
Signaling transport is encrypted. Server sees only session ID + opaque blobs.
DTLS-SRTP on DataChannel
Browser-enforced. Mandatory in all WebRTC implementations.
AES-256-GCM app layer
File data encrypted before entering WebRTC. Server + TURN relay see only ciphertext even if P2P fails.
Why the key never travels over the network
The ECDH exchange works because both sides compute the same secret from different inputs:

Sender computes: sharedSecret = ECDH(senderPrivKey, receiverPubKey)
Receiver computes: sharedSecret = ECDH(receiverPrivKey, senderPubKey)

Mathematics guarantees: both results are identical.

AES key = HKDF-SHA256(sharedSecret, salt=sessionID, info="qrbeam-v1")
The sender's public key travels via QR (air-gapped). The receiver's public key travels via WSS (encrypted). The private keys never leave their respective devices. The shared secret is never transmitted. The signaling server is mathematically blind to the encryption key.
Signaling server — minimal surface area
The signaling server does exactly three things and nothing else:
A
Route join / notify by sessionID
When receiver connects with sessionID, server pings the waiting sender. Passes receiver's pubKey blob opaquely.
B
Relay encrypted SDP + ICE candidates
Passes opaque blobs both ways. Cannot decrypt them (AES-GCM encrypted by app layer). No logging.
C
Self-destruct session after 60s or on DataChannel open
No persistence. In-memory only. Rate-limited: max 10 sessions per IP per minute.
